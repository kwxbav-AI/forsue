# 團隊共用「開發用」雲端資料庫（GitHub 共編）

適合：**多位同事在本機開發，要看同一組營收／出勤／Dashboard 數字**，且專案已在 GitHub 共編。

核心原則：

- **程式碼** → 放 GitHub，大家 `git pull`
- **`DATABASE_URL`（密碼）** → **永不** commit，用 1Password／內部文件分享
- **開發 DB** 與 **正式 DB** 建議分開，避免本機測試誤刪正式資料

---

## 架構建議

```text
GitHub repo（程式碼）
    │
    ├── 正式站 Vercel  →  DATABASE_URL → Neon「production」
    │
    └── 同事本機 npm run dev  →  DATABASE_URL → Neon「development」（共用）
```

| 環境 | 資料庫 | 誰連 |
|------|--------|------|
| 正式站 | Neon **production** | 僅 Vercel Production |
| 本機開發 | Neon **development**（團隊共用） | 所有同事的 `.env` |
| （可選）本機測試 | localhost PostgreSQL | 僅個人實驗，不共用 |

---

## 一、管理員：建立 Neon 開發用資料庫（做一次）

1. 登入 [Neon](https://neon.tech)（若正式站已用 Neon，同一帳號即可）。
2. **New Project**，名稱建議：`daily-performance-dev`（勿與正式專案同名）。
3. 建立後到 **Connection details**，複製 **Connection string**（含 `?sslmode=require`）。
4. 在團隊密碼庫（1Password、Bitwarden、公司 Notion 等）新增一筆，例如：
   - 名稱：`每日績效系統 - DEV DATABASE_URL`
   - 內容：完整連線字串
5. **不要**把這串字貼進 GitHub、Slack 公開頻道或 issue。

### 讓開發庫有資料（擇一）

| 方式 | 說明 |
|------|------|
| **A. 在開發庫重新上傳** | 最安全。同事對 dev 環境上傳與正式相同之營收／出勤檔。 |
| **B. Neon Branch** | 若方案支援，由 production 建立 branch 當 dev（資料與正式相同，仍要小心勿誤改）。 |
| **C. 匯出／匯入** | 管理員 `pg_dump` 正式庫 → `pg_restore` 到 dev（需 DB 操作經驗）。 |

建議新團隊先用 **A**，資料量可控、不會動到正式庫。

### 初始化資料表（管理員執行一次）

在本機暫時把 `.env` 的 `DATABASE_URL` 設成 **dev 連線字串** 後：

```bash
npx prisma generate
npx prisma migrate deploy
npm run db:check
```

確認 `db:check` 可連線（此時營收可能仍為 0，屬正常，待上傳後再驗證）。

## 二、同事：第一次 clone 專案

```bash
git clone <你的 GitHub repo URL>
cd <專案資料夾>
cp .env.example .env
```

1. 向管理員索取 **開發用** `DATABASE_URL`（從 1Password 等），貼入 `.env`：

   ```env
   DATABASE_URL="postgresql://....neon.tech/....?sslmode=require"
   ```

2. 安裝與啟動：

   ```bash
   npm install
   npx prisma generate
   npx prisma migrate deploy
   npm run db:check
   npm run dev
   ```

3. `db:check` 應顯示 **雲端／遠端**，且（若已上傳）營收、出勤筆數 > 0。

4. 開啟 http://localhost:3000 ，用與正式站相同帳號登入（帳號資料也在同一 dev DB）。

詳細除錯見：[本機開發連雲端資料庫.md](./本機開發連雲端資料庫.md)。

## 三、日常共編流程

```bash
git pull
npm install          # package.json 有變才需要
npx prisma migrate deploy   # 有人新增 migration 時必跑
npm run dev
```

- Schema 變更：由改 schema 的同事執行 `npx prisma migrate dev`，再 **push migration 檔案** 到 GitHub。
- 其他同事 `git pull` 後執行 `npx prisma migrate deploy`。
- **不要**把 `.env`、`.env.local` push 上去（已在 `.gitignore`）。

## 四、Vercel / GitHub 與開發庫的關係

| 位置 | 建議設定 |
|------|----------|
| **Vercel Production** | `DATABASE_URL` = Neon **production** |
| **Vercel Preview**（可選） | `DATABASE_URL` = Neon **development**，PR 預覽不碰正式資料 |
| **GitHub Actions Secrets** | 僅 CI 自動化用；**同事本機不從這裡讀**，仍用 1Password 分享 dev 字串 |

同事本機 **不必**從 Vercel 複製，只要用團隊共用的 **dev** 連線即可。

## 六、常見問題

**Q：可以和正式站共用同一個 Neon 資料庫嗎？**  
可以，但本機 `npm run dev` 的刪改、重算會直接影響正式資料，**不建議**。請用獨立 `daily-performance-dev`。

**Q：同事 A 上傳的檔案，同事 B 看得到嗎？**  
可以。共用 dev DB 時，上傳寫入同一資料庫，大家本機重新查詢即可。

**Q：GitHub 上要放什麼？**  
只放程式碼、`prisma/migrations`、`.env.example`（無真密碼）、本文件。

**Q：連線字串洩漏怎麼辦？**  
到 Neon 重設密碼、更新 1Password、通知全員更換 `.env`。
