# 資料庫架構與資料保存說明

本文件說明「每日績效計算系統」目前使用的 **PostgreSQL** 資料結構（以 Prisma `schema.prisma` 為準），以及營收／工時等重要資料**可能消失的原因**與**可改善的保存方式**。

---

## 一、技術棧與連線方式

| 項目 | 說明 |
|------|------|
| 資料庫引擎 | **PostgreSQL** |
| ORM | **Prisma**（`prisma/schema.prisma`） |
| 連線設定 | 環境變數 **`DATABASE_URL`**（應用程式啟動時讀取） |

**重點**：程式本身**不**把資料存在程式碼檔案或 Git 裡；所有「營收、工時、上傳紀錄」都寫在**您指定的那一台 PostgreSQL 資料庫**裡。換成另一個 `DATABASE_URL`（或新開一個空資料庫），就會變成「另一份資料」或「看起來像資料不見了」。

---

## 二、資料表總覽（依業務分類）

### 2.1 門市與主檔

| 模型名稱 | 資料表（概念） | 主要用途 |
|----------|----------------|----------|
| **Store** | `Store` | 門市：代碼、名稱、部門、是否啟用 |
| **StoreAlias** | `StoreAlias` | 門市別名代碼（對應上傳檔的門市代碼） |
| **Employee** | `Employee` | 員工：編號、姓名、預設門市、職位、是否儲備人力、工時比例等 |

---

### 2.2 營收與每日績效（核心計算結果）

| 模型名稱 | 主要欄位／約束 | 說明 |
|----------|----------------|------|
| **RevenueRecord** | `revenueDate` + `storeId` **唯一** | 每日營收（含現金、LINE Pay、支出等）、可關聯上傳批次 |
| **PerformanceDaily** | `workDate` + `storeId` + `versionNo` **唯一** | 每日工效比、總工時、是否達標等**計算後快照** |

> 營收與工時的「原始輸入」與「彙總結果」分開：`RevenueRecord`（營收）、`AttendanceRecord`（出勤工時）等為輸入；`PerformanceDaily` 為依規則重算後的門市日報。

---

### 2.3 出勤、調度、工時異動

| 模型名稱 | 說明 |
|----------|------|
| **AttendanceRecord** | 出勤：日期、員工、原門市、部門、工時、上下班時間、班別等 |
| **DispatchRecord** | 人員調度：起迄日、門市、預申請／實際時數、確認狀態、備註等 |
| **WorkhourAdjustment** | 工時異動調整（類型為列舉，如店長會議、晉升考核等） |

---

### 2.4 上傳與其他

| 模型名稱 | 說明 |
|----------|------|
| **UploadBatch** | 上傳批次紀錄：檔案類型、檔名、筆數、狀態、有效日期等（**不**存 Excel 二進位內容，僅紀錄與檔名） |
| **StoreHourDeduction** | 門市扣工時（效期、清掃、現貨文登記等） |
| **PerformanceTargetSetting** | 目標值設定（含生效區間） |
| **Holiday** | 假日設定（日期唯一） |
| **ContentEntry** | 內容篇數填報與扣工時分鐘數 |
| **AuditLog** | 稽核紀錄（模組、動作、變更前後 JSON） |

---

## 三、關聯關係（摘要）

- **Store** ← 一對多 → **Employee**（`defaultStoreId`）、**RevenueRecord**、**PerformanceDaily**、**StoreAlias**、**StoreHourDeduction**
- **Employee** ← 一對多 → **AttendanceRecord**、**DispatchRecord**、**WorkhourAdjustment**
- **UploadBatch**：以 `uploadBatchId` 選擇性關聯到多種上傳資料（schema 中為可選欄位，依實作寫入）

刪除門市時，**StoreAlias**、**StoreHourDeduction** 設為 `onDelete: Cascade`（連帶刪除）；**Employee** 與 **Store** 的關係未在 schema 中對 `Employee` 做 cascade 刪除說明，實務上仍應避免直接刪主檔，建議以「停用」為主（與門市 `isActive` 設計一致）。

---

## 四、索引與唯一鍵（與查詢／重算有關）

- **RevenueRecord**：`@@unique([revenueDate, storeId])` — 同一日、同一門市一筆營收（upsert 依此鍵）。
- **PerformanceDaily**：`@@unique([workDate, storeId, versionNo])` — 支援版本欄位；重算時會依業務邏輯寫入。
- **AttendanceRecord**、**DispatchRecord** 等：多為 `[workDate]`、`[employeeId, workDate]` 索引，利於報表與區間查詢。

## 五、資料「有可能消失」嗎？常見原因

### 5.1 不是「真的消失」，而是連到別的資料庫

- 本機 `.env`、Render／Vercel 的 **`DATABASE_URL` 不同**（例如一個連 Neon、一個連 Render Postgres、或另一個專案）。
- 部署時**新建**一個空 PostgreSQL，**沒有**把舊資料匯入，也沒跑 `prisma migrate deploy` 到「同一個已有資料的實例」。
- **連線字串**（主機、資料庫名稱、使用者）打錯，連到測試庫或空庫。

**改善**：把「正式環境」的 `DATABASE_URL` 當機密管理，並在文件或內部 wiki 註明**哪一個 URL 是正式庫**；變更前先做備份。

---

### 5.2 應用程式或維運操作造成的資料刪除

- **重匯／覆寫**：上傳邏輯若設計為「先刪除某日再寫入」，同一日期的資料會被替換。
- **Cascade 刪除**：刪除門市等父資料時，子資料可能一併刪除（依 schema 的 `onDelete`）。
- **手動執行 SQL** 或錯誤的 migration 修復。

**改善**：重要操作前**備份**；生產環境限制直接 SQL；上傳功能盡量「可預覽、可還原」或保留歷史批次（目前 `UploadBatch` 有紀錄，但還原仍依賴備份或重新上傳檔案）。

---

### 5.3 雲端資料庫服務本身的風險

- 免費方案或試用到期、**專案被刪除**。
- 區域／服務異常（機率低，但非零）。
- 未開啟**自動備份**或 Point-in-Time Recovery（PITR）。

**改善**：使用**付費層級**、**自動每日備份**、**可下載的備份檔**；重要資料另做週期性匯出（見下節）。

---

## 六、可改善的保存方式（架構建議）

以下為常見做法，可依預算與維運能力選擇；**不論哪一種，核心都是「PostgreSQL + 備份 + 連線設定正確」**。

### 6.1 受管 PostgreSQL（建議）

- **Google Cloud SQL（PostgreSQL）**、**AWS RDS**、**Azure Database for PostgreSQL**、**Neon**、**Supabase**、**Render 內建 Postgres** 等。
- 優點：自動備份、高可用選項、監控、存取控制。
- 應用程式可部署在 **Cloud Run**、**Cloud Run + Cloud SQL**、**Render**、**Vercel** 等，只要 **`DATABASE_URL` 指向同一個穩定實例**、網路與防火牆允許連線即可。

### 6.2 Cloud Run + Cloud SQL（您提到的組合）

- **Cloud Run**：跑 Next.js（容器）無狀態，**不**負責資料持久化。
- **Cloud SQL**：PostgreSQL 真正存資料；透過 **Cloud SQL Auth Proxy** 或 **Unix socket / private IP** 連線。
- 適合：**資料與運算分離**、需要與 GCP 其他服務整合、或需要固定區域與合規。

### 6.3 應用程式層的補強（非取代資料庫）

- **定期匯出**：排程將關鍵表（或全庫 `pg_dump`）匯出到 **Object Storage**（GCS/S3）與保留政策。
- **只讀副本**：報表大量查詢可接副本，減輕主庫壓力（進階）。
- **Prisma Migrate**：schema 變更時用 migration，**避免**手改正式庫結構與程式碼不同步。

## 八、簡短結論

| 問題 | 說明 |
|------|------|
| 目前架構是什麼？ | **PostgreSQL**，多張業務表；營收在 `RevenueRecord`，工時在 `AttendanceRecord` 等，計算結果在 `PerformanceDaily`；詳見上表。 |
| 資料會不會消失？ | **會**，若連錯資料庫、覆寫匯入、刪除父資料、或雲端專案被刪／無備份。 |
| 怎麼改善？ | **固定正式庫的 `DATABASE_URL`**、**受管 DB + 自動備份**、**定期 `pg_dump` 或匯出**；**Cloud Run + Cloud SQL** 是「運算與資料庫分離」的穩健組合之一，與 Neon／Render Postgres 等**同屬**「把 PostgreSQL 當唯一真相來源」的思路。 |
